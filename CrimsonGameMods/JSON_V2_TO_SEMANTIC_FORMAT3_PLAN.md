# JSON v2 to Semantic Format 3 Plan

## Summary

Yes, this is possible, but the first safe target should be **JSON v2 iteminfo mods only**. The Stacker already has most of the needed model: JSON v2 ingestion, reparse-diff `FieldEdit`s, semantic apply, and field-level merge. The work is to add a clean export/translation layer that turns those recovered `FieldEdit`s into format `3` intents.

Important blocker found during planning: this local `crimson_rs.pyd` is missing `inspect_legacy_patches`, and its current `parse_iteminfo_from_bytes` failed against local vanilla iteminfo. The implementation must first align/update the bundled parser runtime or make the translator clearly disabled with a useful error.

## Key Changes

- Add a **Format 3 semantic JSON model**:
  - Top-level: `modinfo`, `format: 3`, `target: "iteminfo.pabgb"`, `intents`.
  - Intent fields: `entry`, `field`, `op`, `new`, optional `label`, optional `source_patches`.
  - Supported ops map directly from existing `FieldEdit`s:
    - `SET` -> `op: "set"`
    - `APPEND` -> `op: "append"`
    - `REMOVE_IDX` -> `op: "remove_indices"`
    - `ADD_ENTRY` -> `op: "add_entry"`
    - `REMOVE_ENTRY` -> `op: "remove_entry"`

- Add a translator service in the Stacker/inspector layer:
  - Input: legacy JSON v2 file plus vanilla `0008/iteminfo.pabgb`.
  - Use `reparse_diff_patches()` as the authoritative safe translator.
  - Convert only successfully recovered `FieldEdit`s into format 3 intents.
  - Write skipped/untranslated patches to a conversion report instead of guessing.
  - If `inspect_legacy_patches` is available, use it only for richer Details labels and field attribution.
  - If `inspect_legacy_patches` is unavailable, do not disable conversion as long as full parse + reparse-diff works.

- Add Stacker UI actions:
  - For a selected `legacy_json` source: `Export Semantic JSON`.
  - After Preview/Apply produces merged items: `Export Stack as Semantic JSON`, generated by diffing vanilla parsed items vs merged parsed items.
  - Show counts in Details: translated intents, skipped patches, parse-break patches, no-op patches.

- Add format 3 import support to Stacker:
  - Detect `format: 3`, `target: "iteminfo.pabgb"`.
  - Apply intents to parsed vanilla dicts using the existing field-edit apply path.
  - Merge format 3 mods field-by-field with existing sources.

## Test Plan

- Parser/runtime prerequisite:
  - Verify bundled `crimson_rs` can parse vanilla `0008/iteminfo.pabgb`.
  - Verify either `inspect_legacy_patches` exists or translator uses reparse-diff without requiring it.
  - If parsing is complete, verify JSON v2 conversion no longer reports the parser/runtime blocker.

- Example conversions:
  - `AccessorySocketsMod.json`: emits restriction `set` intents and socket-list `append` intents.
  - `Universal Abyss Gear Unlock.json`: emits `broken_item_prefix_string` `set` intents.
  - `More Inventory-JSON.json`: reports unsupported target `inventory.pabgb` for this MVP, without crashing.

- Roundtrip validation:
  - Convert JSON v2 -> format 3.
  - Apply format 3 intents to vanilla parsed iteminfo.
  - Compare resulting parsed dict changes against the current Stacker reparse-diff output.

- UI validation:
  - Selected JSON v2 source can export semantic JSON.
  - Export reports skipped/unsupported patches clearly.
  - Format 3 JSON can be dropped back into Stacker and merged like other sources.

## Assumptions

- MVP scope is **JSON v2 iteminfo mods first**.
- Conversion policy is **safe intents only**: uncertain/unsupported byte patches are reported, not guessed.
- Inventory mods like `More Inventory-JSON.json` should be handled later by a separate `inventory.pabgb` semantic translator, not forced into the iteminfo format.
